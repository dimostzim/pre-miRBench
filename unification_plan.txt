UNIFICATION PLAN — Unified probability CSV output for all 6 tools
Goal: each tool emits predictions.csv with exactly two columns:
        window_id        — FASTA record ID (string, as written in the input .fa header)
        probability_score — float [0, 1], probability that the window is a pre-miRNA
This makes AUC computation trivial and removes the bespoke parsing logic in tool_adapters.py.
No model weights, model architectures, or training code are touched.
================================================================================


BACKGROUND: WHY THE THREE BINARY TOOLS NEED SOURCE PATCHES
-----------------------------------------------------------
dnnpremir, deepmirgene, and deepmir all use a 2-class softmax as their final layer.
After the forward pass the raw output is a (n, 2) float matrix — a genuine probability
distribution. Each tool immediately discards it with np.argmax and writes a binary label.
There is no threshold tuning, no calibration, no post-processing: it is a plain argmax.
The probability was always there; it was just never persisted to disk.

To recover it we patch the upstream source files at Docker build time (the same mechanism
already used in deepmirgene/patch_upstream.py for Keras compatibility fixes). No model
is changed — only the final file-writing block.

CHANGE 1 — mire2e
================================================================================
File to change: tools/mire2e/inference.py

Current behaviour
  For every FASTA record, model.predict() is called and returns:
    scores_5_3  : array of floats, one per internal sub-window
    scores_3_5  : array of floats, one per internal sub-window
    index       : list of sub-window IDs, each formatted as "<record_id>-<start>-<end>"
  All entries are collected into a flat list and written as predictions.json.
  The benchmark then uses _infer_mire2e_record_id() to strip the trailing -start-end
  suffix and recover the original record_id, taking max(score_5_3, score_3_5) per record.

What to change
  Perform that aggregation inside inference.py instead of deferring it to tool_adapters.
  After the loop over FASTA records, group all sub-window entries by parent record_id
  (strip the trailing "-<int>-<int>" suffix from each index value), take
  max(score_5_3, score_3_5) across all sub-windows belonging to that record, and write
  one row per original FASTA record.

  New output: predictions.csv (replaces predictions.json)
    Header:  window_id,probability_score
    Example: balanced_benchmark__000001,0.9821

Implementation sketch (inference.py changes only):
  - Replace the `predictions` list with a dict: best_score = {}
  - After model.predict() for each record, iterate over (index, scores_5_3, scores_3_5):
        parent_id = idx.rsplit("-", 2)[0]   # strip sub-window suffix
        score = max(float(s53), float(s35))
        if score > best_score.get(parent_id, 0.0):
            best_score[parent_id] = score
  - After all records: write predictions.csv from best_score dict
  - Remove the JSON import and output_file logic

Why rsplit("-", 2)[0] is safe: the record_id format in the benchmark is
"balanced_benchmark__NNNNNN" (no trailing hyphens), and mire2e appends -<start>-<end>,
so stripping two hyphen-delimited tokens from the right is unambiguous.

Side effect: predictions.json is no longer produced. RESULT_FILES in tool_adapters.py
must be updated (see Change 7).

CHANGE 6 — mustard
================================================================================
File to change: tools/mustard/inference.py

Current behaviour
  Runs MuStARD.pl which writes a gzipped TSV at:
    predict/static/results/intermediate_files/<prefix>.predictions.txt.gz
  Format: one tab-separated line per interval, two float columns (class_0, class_1),
  no header, no sequence IDs — purely positional relative to the input BED file.
  inference.py does no post-processing; the benchmark reads the .gz file directly
  and aligns by position to the metadata file.

What to change
  Add post-processing inside inference.py after subprocess.check_call(cmd) returns.

  Steps:
  1. Read the BED file (args.targetIntervals) to extract record IDs in order.
     BED column 3 (0-indexed) is the record_id as written by write_tool_inputs().
  2. Glob for the single *.predictions.txt.gz produced by MuStARD.pl under args.dir.
  3. Read each line of the gzipped TSV: two tab-separated floats per line.
     Column 1 (0-indexed) is the pre-miRNA score (positive class).
  4. Zip record IDs with scores and write predictions.csv.

  New output: predictions.csv in args.dir
    Header:  window_id,probability_score
    Example: balanced_benchmark__000001,0.9877

  The gzipped raw file is kept untouched (MuStARD.pl produced it; we just read it).

Implementation sketch (new block at end of inference.py, inside main(), after
subprocess.check_call(cmd)):

    import gzip, csv as _csv, glob as _glob

    # Read record IDs from BED file in order
    record_ids = []
    with open(args.targetIntervals) as bed_fh:
        for line in bed_fh:
            parts = line.rstrip("\n").split("\t")
            if len(parts) >= 4:
                record_ids.append(parts[3])

    # Find the predictions file MuStARD wrote
    pattern = os.path.join(args.dir, "predict", "static", "results",
                           "intermediate_files", "*.predictions.txt.gz")
    matches = sorted(_glob.glob(pattern))
    if len(matches) != 1:
        raise FileNotFoundError(f"Expected one MuStARD predictions file, found {len(matches)}")
    pred_gz = matches[0]

    # Read scores
    scores = []
    with gzip.open(pred_gz, "rt") as gz_fh:
        for raw_line in gz_fh:
            line = raw_line.strip()
            if line:
                cols = [float(v) for v in line.split("\t") if v]
                scores.append(cols[1])   # col 1 = pre-miRNA probability

    if len(scores) != len(record_ids):
        raise RuntimeError(f"mustard ID/score count mismatch: {len(record_ids)} IDs vs {len(scores)} scores")

    csv_path = os.path.join(args.dir, "predictions.csv")
    with open(csv_path, "w", newline="") as csv_fh:
        w = _csv.writer(csv_fh)
        w.writerow(["window_id", "probability_score"])
        for rid, score in zip(record_ids, scores):
            w.writerow([rid, score])

CHANGE 7 — benchmark/balanced_benchmark/tool_adapters.py
================================================================================
Once all six tools emit predictions.csv with the same header, the five bespoke
normalize_* functions and the RESULT_FILES dict can be replaced by a single generic
reader.

Changes needed:

  RESULT_FILES dict — update/simplify:
    All tools now produce "predictions.csv", so the dict can become:
        RESULT_FILES = {tool: "predictions.csv" for tool in TOOLS}

  resolve_result_path() — remove the mustard special case:
    The glob for the .gz file is no longer needed. The unified path is:
        Path(results_dir) / tool / output_name / "predictions.csv"

  Replace all normalize_* functions with a single function:
    def normalize_unified(output_path, metadata_by_record, threshold):
        rows = []
        with open(output_path, newline="") as handle:
            for row in csv.DictReader(handle):
                record_id = row["window_id"]
                score = float(row["probability_score"])
                rows.append({
                    "record_id": record_id,
                    "window_id": metadata_by_record[record_id]["window_id"],
                    "score": score,
                    "predicted_class": 1 if score >= threshold else 0,
                    "ground_truth_class": gt_to_int(metadata_by_record[record_id]["label"]),
                })
        if len(rows) != len(metadata_by_record):
            raise RuntimeError(f"{output_path}: {len(rows)} predictions vs {len(metadata_by_record)} expected")
        return rows

  normalize_tool_output() — simplify dispatch:
    All branches collapse to a single call to normalize_unified().
    The mustard_positive_column parameter can be removed from the signature.

  FIXED_LENGTHS — no change needed (used in prepare_record_for_tool, not parsing).

RISK / GOTCHAS PER TOOL
================================================================================

mire2e
  - The rsplit("-", 2)[0] trick for stripping sub-window suffix is already validated
    in the existing _infer_mire2e_record_id(). If any record_id itself contains
    hyphens (unlikely given benchmark naming), verify the stripping logic.
  - Taking max(score_5_3, score_3_5) is the same aggregation the benchmark already
    uses, so this does not change effective behaviour, only where it happens.

mirdnn
  - Trivial change. Only risk: upstream mirdnn_eval.py is sometimes updated;
    confirm the column order (id, score) has not changed if rebuilding the image.

dnnpremir
  - The existing patch pattern (replace_once) will fail if the upstream isPreMiR.py
    is updated and the target string changes. Pin the git clone to a specific commit
    in the Dockerfile (add --branch or --single-branch with a known SHA) to prevent
    silent breakage.
  - name_list entries from the RNAfold output file include the full FASTA header
    line with newline; stripping and lstrip(">") is required.

deepmir
  - deepmir's inference.py already does a shutil.copy2 of everything in user_data/,
    so the renamed predictions.csv will be copied to args.output automatically.
  - The runtime_predictor.py reference copy must be kept in sync manually — it is
    not used by Docker but may be used in conda mode.
  - Pin the jacordero/deepmir git clone to a commit SHA in the Dockerfile.

deepmirgene
  - Patch B re-routes through encode() directly instead of import_data(). Confirm
    that import_data() is not called elsewhere in the file; if it is, update those
    call sites too or keep import_data() working by having it call the new two-return
    import_seq() properly.
  - predictions[:, 0] is pre-miRNA probability (confirmed: class 0 = pre-miRNA in the
    original paper and in the class_label == 0 check in the source).

mustard
  - The post-processing code must run AFTER MuStARD.pl has fully written the .gz file.
    subprocess.check_call() already blocks until completion, so timing is fine.
  - If staticPredFlag=0 (sliding window mode) is ever used, the number of output rows
    will not match the number of BED intervals. The plan only covers staticPredFlag=1
    (static mode), which is what the benchmark uses.
